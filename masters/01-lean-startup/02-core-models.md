> Build-Measure-Learn 循环与 MVP 体系

## 核心循环：Build-Measure-Learn

### 循环详解

```
┌─────────────────────────────────────────────────────────────────┐
│                         BUILD 阶段                              │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 制作最小可行产品 (MVP)                                    │  │
│  │ - 只实现核心功能                                          │  │
│  │ - 能验证关键假设                                          │  │
│  │ - 快速低成本上线                                          │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                         MEASURE 阶段                            │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 收集真实用户数据                                          │  │
│  │ - 行为数据（analytics）                                   │  │
│  │ - 定性反馈（用户访谈）                                    │  │
│  │ - 可执行指标 vs 虚荣指标                                  │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                         LEARN 阶段                              │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 从数据中提取洞察                                          │  │
│  │ - 假设验证成功 → 坚持方向                                │  │
│  │ - 假设验证失败 → 转向（Pivot）                            │  │
│  │ - 持续减少不确定性                                        │  │
│  └───────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              └──────────────────┐
                                               │
                              ┌────────────────▼───────────────┐
                              │      循环迭代，持续优化          │
                              └─────────────────────────────────┘
```

### 关键原则

```
1. 速度优先
   - 快速进入循环
   - 每次循环都有学习
   - 迭代周期越短越好

2. 数据驱动
   - 用真实数据验证假设
   - 避免主观判断
   - 定量 + 定性结合

3. 假设导向
   - 每个实验都有明确假设
   - 验证成功/失败标准前置
   - 学习是持续过程
```

## MVP 体系

### MVP 的三种类型

```
类型一：单功能MVP
┌────────────────────────────────────┐
│  功能A  |  功能B  |  功能C          │
│   ✓     |   ✗    |   ✗              │
│  只做核心功能，其他跳过              │
└────────────────────────────────────┘

类型二：线下MVP
┌────────────────────────────────────┐
│  在产品化之前，先用手工方式验证       │
│  例：Airbnb创始人亲自拍照           │
└────────────────────────────────────┘

类型三：贵宾MVP
┌────────────────────────────────────┐
│  1个用户 × 完整功能 = 贵宾服务       │
│  服务好1个人，验证完整价值主张       │
└────────────────────────────────────┘
```

### MVP vs 完整产品

```
┌────────────────┬────────────────┬────────────────┐
│      维度      │     MVP        │    完整产品    │
├────────────────┼────────────────┼────────────────┤
│    目标        │ 验证假设        │ 占领市场        │
├────────────────┼────────────────┼────────────────┤
│    功能        │ 核心功能        │ 完整功能        │
├────────────────┼────────────────┼────────────────┤
│    时间        │ 最短时间        │ 充足时间        │
├────────────────┼────────────────┼────────────────┤
│    成本        │ 最小成本        │ 完整投入        │
├────────────────┼────────────────┼────────────────┤
│    风险        │ 低风险          │ 高风险          │
├────────────────┼────────────────┼────────────────┤
│    学习        │ 高学习          │ 低学习          │
└────────────────┴────────────────┴────────────────┘
```

## 验证性学习框架

### 如何验证假设

```
假设验证矩阵：

假设类型          验证方法              成功标准
─────────────────────────────────────────────────
用户需求存在      行为数据              用户主动使用
用户愿意付费      转化数据              完成支付
用户会推荐       NPS/推荐率            NPS > 50
留存会持续       留存曲线              Day30 > 20%
```

### 五种转向（Pivot）

```
1. 放大转向 (Zoom-in Pivot)
   从一个功能变成整个产品
   "这个功能才是产品"

2. 缩小转向 (Zoom-out Pivot)
   从一个产品变成更大产品的一部分
   "这个产品太窄了"

3. 客户细分转向 (Customer Segment Pivot)
   换一批目标用户
   "不是这群人"

4. 渠道转向 (Channel Pivot)
   通过不同渠道触达同样用户
   "换一批人找到这批用户"

5. 技术转向 (Technology Pivot)
   用不同技术达到同样效果
   "换一种方式实现"
```

## 创新核算

### 三步验证流程

```
Step 1: 建立基准线
┌─────────────────────────────────────────────────┐
│                                                 │
│   MVP上线后，测量关键指标作为基准                 │
│                                                 │
│   例：新用户激活率 = 5%                          │
│       用户留存率（Day7） = 20%                   │
│                                                 │
└─────────────────────────────────────────────────┘

Step 2: 改善引擎
┌─────────────────────────────────────────────────┐
│                                                 │
│   通过实验改善核心指标                           │
│                                                 │
│   实验A：优化注册流程     → 激活率 5% → 8%      │
│   实验B：优化激活流程     → 激活率 8% → 12%     │
│   实验C：新用户引导       → 激活率 12% → 15%    │
│                                                 │
└─────────────────────────────────────────────────┘

Step 3: 决策：坚持 or 转向
┌─────────────────────────────────────────────────┐
│                                                 │
│   数据支持继续 → 坚持 (Persevere)               │
│   数据支持改变 → 转向 (Pivot)                   │
│                                                 │
│   判断标准：                                     │
│   - 指标是否在改善？                            │
│   - 改善速度是否足够？                          │
│   - 方向是否正确？                              │
│                                                 │
└─────────────────────────────────────────────────┘
```

###  vanity metrics vs 可执行指标

```
❌ 避免的虚荣指标：
   - "我们有100万用户"
   - "我们的App被下载了500万次"
   - "我们的页面有1000万浏览"

✅ 关注的可执行指标：
   - 激活率：完成关键行为的用户比例
   - 留存率：持续使用产品的用户比例
   - 转化率：完成目标价值的用户比例
   - 病毒系数：每个用户带来多少新用户
```

## 实战工具

### 实验设计模板

```
┌─────────────────────────────────────────────────────────────┐
│                      实验设计文档                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  实验名称：[清晰描述实验目的]                                 │
│                                                             │
│  假设：                                                      │
│  我们相信 [用户群体] 在 [场景] 下会 [行为]                    │
│  因为 [证据/数据/观察]                                       │
│                                                             │
│  实验组：                                                    │
│  [具体改动内容]                                             │
│                                                             │
│  对照组：                                                    │
│  [保持不变的内容]                                           │
│                                                             │
│  成功标准：                                                  │
│  [可执行指标] 提升 [X]%，统计显著 (p < 0.05)                 │
│                                                             │
│  最少样本量：[计算所需样本]                                   │
│  实验周期：[时间长度]                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### ICE 优先级框架

```
ICE = Interest × Confidence × Ease

Interest（兴趣）
   这个实验对用户有多重要？
   - 10分：用户会明显感受到
   - 5分：用户可能注意到
   - 1分：用户不会注意到

Confidence（信心）
   你对这个假设有多大把握？
   - 10分：有数据支持
   - 5分：基于经验
   - 1分：纯猜测

Ease（易度）
   实现这个实验需要多少工作？
   - 10分：1天完成
   - 5分：1周完成
   - 1分：1个月完成

优先选择 ICE 分数最高的实验
```
